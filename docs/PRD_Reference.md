# 虚拟商品代充值产品全景指南（PRD Reference）

本文件旨在为开源贡献者、集成开发者、产品经理提供清晰的业务背景，帮助大家理解 **recharge-agent-skill** 这个 AI Agent 插件背后的真实商业闭环。

---

## 一、项目背景

**Jiigo × 彩贝壳** 联合打造的「虚拟商品代充值」业务，核心解决两类用户痛点：

- **买家**：想快速、安全地为游戏、视频、直播等虚拟商品充值，但担心充错账号、价格不透明、售后困难。
- **卖家（个人代充手）**：希望通过副业获得稳定收入，但面临接单效率低、资金安全无保障、履约时效难控制等问题。

平台通过 **交易担保 + AI Agent 智能引导 + 卖家侧 AI 助手** 三层机制，实现「资金先托管、履约后结算」的闭环，极大提升了交易信任与效率。

---

## 二、双边市场角色定位

### 1. 买家侧（彩贝壳 / Jiigo 买家域）

AI Agent 扮演「智能充值助手」：

- 交互式收集充值需求
- 严格执行账号格式校验
- 推荐全网最优价格
- 强制「红灯泡二次确认」
- 引导拉起安全收银台
- 支付后持续监控履约状态

### 2. 卖家侧（Jiigo AI 副业助手）

AI Agent 扮演「履约提醒与派单助手」：

- 接收系统派单通知
- 提醒个人代充手在 **15 分钟内** 完成接单、充值、上传凭证
- 超时自动触发全额退款
- 发货后协助买家确认收货流程

## 三、核心业务规则（必须严格遵守）

### 1. 级联 SKU 结构

每个品牌都有严格的**两级级联规格**：

- **一级规格（level1Name）**：例如「白金会员」「黄金会员」「抖音币」
- **二级规格（durationOrAmount）**：例如「3个月」「100抖币」

每条二级规格都实时返回当前全平台最优价格 `bestPrice`。

**AI Agent 必须严格按此顺序引导用户**：
1. 先选一级规格
2. 再选二级规格
3. 主动告知 `bestPrice`

### 2. 账号格式校验（红灯泡前置）

- 每个品牌模板都定义了 `inputFieldName`（例如「充值账号(手机号)」）
- 以及对应的正则 `inputRegex`（例如 `^1[3-9]\d{9}$`）
- **AI Agent 在调用下单接口前，必须先用正则本地校验**
- 校验失败立即拦截

### 3. 红灯泡二次确认（重中之重）

在真正调用 `createRechargeOrder` 之前，AI **必须** 输出以下高亮确认话术：

```
🚨【重要账号安全确认】🚨
请您仔细核对以下充值信息：
------------------------------------
充值账号：[用户输入的账号] (请确保无错，充错无法退款)
充值规格：[品牌]·[充值类型]·[有效期/数量]
应付金额：￥[推荐价]
------------------------------------
确认无误请回复“确认”，我将为您唤起安全支付收银台。
```

**只有用户明确回复「确认」或表达肯定意向后，才能调用下单接口。**

### 4. 单次限购 1 件

- `quantity` 字段**必须恒等于 1**
- 用户尝试购买多份（如“帮我充 2 个月”）时，必须委婉拒绝：
  > “为了确保代充账户的安全与充值成功率，每次仅支持下单购买 1 份，请分次购买。”

### 5. 15 分钟发货大闸

- 买家支付成功后，订单进入履约流程
- **卖家必须在 15 分钟内** 完成「接单 + 充值 + 上传凭证」
- 超时未发货 → 系统**自动全额退款**（状态变为 `REFUNDED`）

### 6. 次日自动收货

- 卖家发货后（状态 `DELIVERED`），会返回 `rechargeProofUrl`（充值凭证截图）
- 若买家未手动确认收货，系统将在**发货次日 23:59:59** 自动确认收货并将资金结算给卖家
- AI Agent 应在接近该时间时提醒用户及时确认

## 四、订单状态机

```
PENDING_PAY 
   ↓ (支付成功)
PENDING_MATCH 
   ↓ (匹配卖家)
PENDING_ACCEPT 
   ↓ (卖家接单)
RECHARGING 
   ↓ (充值中)
DELIVERED (已发货，附带 rechargeProofUrl)
   ↓ (买家确认或自动确认)
COMPLETED

异常分支：
- 超时未履约 → REFUNDED（全额退款）
- 其他异常 → CLOSED
```

## 五、关键安全与合规机制

| 机制               | 实现位置                  | 目的                     |
|--------------------|---------------------------|--------------------------|
| 账号正则校验       | `getBrandTemplate` + Prompt | 防止充错账号             |
| 红灯泡二次确认     | System Prompt 强制输出    | 用户肉眼核对，平台免责   |
| Quantity=1 硬限制  | OpenAPI + 后端校验        | 防刷、降低风控           |
| X-Idempotency-Key  | `/orders/create` 请求头   | 防止重复扣款             |
| 15 分钟自动退款    | 后端异步定时器            | 保护买家资金             |
| 凭证图 + 自动收货  | `OrderStatusResponse`     | 保障卖家及时回款         |

## 六、AI Agent 在闭环中的职责

### 买家侧 Agent 必须做到的行为约束

1. 表达充值意向 → 立即调用 `listBrands`
2. 选定品牌 → 立即调用 `getBrandTemplate`
3. 严格按级联顺序收集 `skuLevel1` → `skuLevel2`
4. 收集账号后，先用 `inputRegex` 校验
5. 下单前**必须**展示红灯泡确认框
6. 用户明确确认后，才调用 `createRechargeOrder`
7. 支付成功后开始轮询 `getOrderStatus`
8. 状态变为 `DELIVERED` 时，主动展示 `rechargeProofUrl` 图片
9. 接近自动收货时间时提醒用户确认

### 禁止行为

- 跳过 `getBrandTemplate` 直接下单
- 自行构造不存在的 SKU 组合
- 修改 `bestPrice` 或 `quantity`
- 未做二次确认就调用创建订单接口
- 忽略超时退款或自动收货规则

## 八、典型用户旅程（买家视角）

1. 用户对 AI 说：「我想充腾讯视频白金会员 3 个月」
2. AI 调用 `listBrands` → 确认品牌
3. AI 调用 `getBrandTemplate` → 拿到级联规格和价格
4. 引导用户选择 → 白金会员 → 3个月 → 告知 ¥79.9
5. 询问充值手机号 → 校验正则通过
6. 输出红灯泡确认框
7. 用户回复「确认」
8. 调用 `createRechargeOrder` → 返回支付链接
9. 用户支付 → AI 模拟支付成功 → 启动 15 分钟计时器
10. 卖家在 10 分钟内发货 → AI 轮询到 `DELIVERED` → 展示凭证截图
11. 用户确认收货 → 订单 `COMPLETED`
